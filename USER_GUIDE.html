01 Overview

Understanding the SCL-Governor and the autonomous control loop architecture.

What is SCL-Governor?

SCL-Governor is an autonomous infrastructure management system that uses an LLM (Large Language Model) as the supervisory intelligence for a continuous control loop. It monitors your applications and infrastructure in real time, predicts issues before they happen, simulates corrective actions, and makes intelligent decisions about whether to act autonomously or involve a human.

Think of it as a tireless SRE that watches your systems 24/7, reasons about their health, and takes the best available action at every moment — all while learning from the outcomes to get better over time.

←←← continuous feedback loop ←←←
  1. Observe — Pull telemetry from Prometheus across 5 signal classes (infrastructure, application, business, network, cost). Detect anomalies, compute trend vectors, and determine the system regime.
  2. Predict — Use statistical models and LLM reasoning to forecast multi-horizon risks: SLA breach probability, cascading failure risk, and cost overruns.
  3. Simulate — Run Monte Carlo simulations of candidate actions. Compute Pareto frontiers across cost, performance, and risk. Calculate VaR and CVaR metrics.
  4. Decide — The LLM synthesizes observations, predictions, and simulations to select the optimal action and assign an autonomy level based on confidence.
  5. Actuate — Execute the chosen action (scaling, rollback, config change, etc.) via Kubernetes or other actuators. Send notifications via Slack or PagerDuty.
  6. Learn — Track prediction accuracy, simulation fidelity, and reward signals. Detect model drift and store experience in the replay buffer.

02 Quick Start

Get SCL-Governor running in under 5 minutes.

Clone and navigate to the project

git clone <repository-url>
cd LLM-Supervised-Control-Loops-SCL

Copy the environment template

cp .env.example .env

Configure your API keys

Open .env and set your LLM provider credentials:

# Choose one:
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
Tip

You can skip this step to start in demo mode with synthetic data. You can add your API key later via the Connections page.

Run setup and start development servers

# Install dependencies and start locally
make setup
make dev

# Or use Docker Compose for the full stack
make up

Docker Compose will start the backend, frontend, Prometheus, and Grafana together.

Open the dashboard

Navigate to http://localhost:3000 in your browser. The API docs are at http://localhost:8000/docs.

Onboard your application

Go to the Connections page (via the sidebar) to connect your real Prometheus instance and configure monitoring for your services. See the next section for a detailed walkthrough.

Understanding the Metrics

  • Expected Objective — The weighted multi-objective score across performance, cost, risk, stability, and business dimensions. Higher is better.
  • VaR (Value at Risk) — The worst-case objective value at the 5th percentile. Shows the downside risk of an action.
  • CVaR (Conditional VaR) — The average objective in the worst 5% of scenarios. A more conservative risk measure than VaR.
  • SLA Breach % — Percentage of simulated scenarios where the action leads to an SLA violation.
  • Pareto Optimal — An action is Pareto-optimal if no other action is better in all dimensions simultaneously. These are the best candidates.

09 Decisions

Audit log, autonomy classification, and human override interface.

Every decision made by SCL-Governor is logged with full context: the selected action, reasoning, confidence score, autonomy level, alternative actions considered, and rollback plan. This page provides complete audit trail transparency.

Autonomy Levels

The autonomy level is determined by the decision confidence score and the thresholds configured in Settings:

Execute Autonomous

Confidence above the "high" threshold (default 85%). The governor executes the action immediately without human involvement. Best for well-understood, low-risk, reversible actions.

Execute with Notification

Confidence between "medium" and "high" thresholds. The action is executed and a notification is sent to the team via Slack or PagerDuty. No human approval needed, but team is informed.

Recommend

Confidence between "low" and "medium" thresholds. The governor recommends an action but does not execute it. A human must review and approve via the override interface.

Escalate

Confidence below the "low" threshold, or extremely high risk. The governor escalates to on-call via PagerDuty and takes no action. Typically triggered during novel failure modes.

Human Override

Click the Human Override button in the top-right to manually submit an action. Enter the desired action (e.g., "scale_up api-service to 5 replicas") and a reason. The override is logged and the governor incorporates human feedback into its learning loop.

Decision Detail View

Click any row in the decision table to expand it and see the full reasoning, rollback plan, rollback trigger conditions, blast radius, reversibility score, cost delta, and number of alternative actions considered.

What is Tracked

  • Prediction Accuracy — How well the system's predictions matched actual outcomes. Shown as an area chart trending over cycles.
  • Simulation Fidelity — How closely Monte Carlo simulation results matched real-world outcomes after action execution.
  • Reward Signal — A composite score reflecting whether the actions taken improved the system state. Positive rewards indicate improvement.
  • Model Drift — A percentage indicating how much the underlying data distribution has shifted. Values above 10% suggest the models should be retrained.
  • Human Overrides — Count of manual interventions. Each override is used as a learning signal to improve future decisions.
  • Replay Buffer — The number of state-action-reward transitions stored for offline learning and model updates.

Configuration Details

  • Cycle Interval — How often the control loop runs (1s to 60s). Lower values mean faster response but higher resource usage. Default: 15 seconds.
  • Objective Weights — Balance the multi-objective optimization. Weights should sum to 1.0. Increase "performance" weight to prioritize speed; increase "cost" to prioritize savings.
  • Confidence Thresholds — Define the boundaries between autonomy levels. High (default 85%): autonomous execution. Medium (65%): execute with notification. Low (40%): recommend only.
  • Safety Constraints — Hard limits the governor will never violate: minimum replica count, budget ceiling, maximum blast radius for any single action, maximum concurrent changes, and cooldown between actions.
  • Connector Status — Shows which external integrations (Prometheus, Kubernetes, LLM, Slack, PagerDuty) are currently connected and operational.

13 Troubleshooting & FAQ

Common questions and solutions for typical issues.

I see "Demo Mode" -- how do I connect real infrastructure?

Navigate to the Connections page and click Onboard New Application. Enter your Prometheus URL, add your services with SLO targets, and click Save & Activate. The governor will switch from synthetic data to real telemetry immediately. See Section 3: Onboarding for the full walkthrough.

The governor shows "critical" regime by default

This is normal when running in demo mode with synthetic telemetry data. The demo generator produces varying conditions including degraded and critical regimes to showcase the full range of system behavior. Once you connect real infrastructure, the regime will reflect your actual system health.

LLM reasoning is not working or shows generic responses

Make sure you have configured an API key for your LLM provider. You can do this in two ways:

  • Set ANTHROPIC_API_KEY or OPENAI_API_KEY in your .env file and restart the backend.
  • Add the API key through the Connections page in the LLM Configuration step when onboarding your application.

Verify the key is valid by using the Test Connection feature on the Connections page.

How do I connect my Kubernetes cluster?

When creating or editing a connection, enable the Kubernetes toggle in Step 4 of the onboarding wizard. Provide either:

  • A kubeconfig path (for external access)
  • Enable "In-Cluster" mode (when SCL-Governor is running inside the same K8s cluster)

Specify the target namespace and optional context name. SCL-Governor needs RBAC permissions for deployments, pods, and horizontal pod autoscalers.

Can I monitor multiple applications?

Yes. Create multiple connections on the Connections page, each with its own Prometheus source, services, and configuration. You can activate different connections to switch the governor's focus. Each connection maintains its own SLO targets and service list.

The frontend is not connecting to the backend

Verify the backend is running at http://localhost:8000. Check the browser developer console for CORS errors. If using Docker Compose (make up), all services should be networked automatically. If running with make dev, ensure both the backend (port 8000) and frontend (port 5173 or 3000) are started.

How do I adjust how aggressively the governor acts?

Go to Settings and adjust two things:

  • Confidence Thresholds — Raise the "high" threshold to require more certainty before autonomous action.
  • Safety Constraints — Lower the max blast radius and max concurrent changes to limit the scope of any single action.
  • Cycle Interval — Increase the cycle interval to give the system more time between decisions.
Where are the logs?

Backend logs are written to stdout in structured JSON format. When using Docker, view them with make logs or make logs-backend. The backend uses structured logging with context fields for easy filtering and analysis.
